## ТЕХНИЧЕСКАЯ РЕАЛИЗАЦИЯ

### Парсинг Telegram-чатов
Библиотека **Telethon** (Python) — читает чаты через Telegram API.
Режимы: полный (первый раз), инкрементальный (ежедневно), по запросу.
Хранение: PostgreSQL, таблица `telegram_messages`.

**Ограничение:** только групповые чаты, личные переписки не парсятся.

### Обработка Zoom-встреч
**Схема A (основная):** Zoom → MyMeet → транскрипты → PostgreSQL (`mymeet.meetings`)
**Схема B (резервная):** Zoom запись → Whisper (локальная транскрибация)

### RAG — семантический поиск
Векторизация через `paraphrase-multilingual-MiniLM-L12-v2` (поддерживает русский).
Хранение векторов в PostgreSQL (pgvector).
Принцип: вопрос → похожие фрагменты → Claude отвечает на реальных данных.

Реализованные скрипты:
- `telegram_rag.py` — индексирование чатов
- `zoom_rag_utils.py` — индексирование транскриптов
- `rag_manager.py` — управление диалогами
- `rag_chunk_meetings.py` — чанкинг встреч

### Стек технологий
- **OpenClaw** — оркестрация агентов, управление ботами
- **Claude (Anthropic)** — языковая модель для анализа
- **PostgreSQL + pgvector** — база данных + векторный поиск
- **Telethon** — парсинг Telegram
- **Python** — скрипты сбора и обработки
- **VPS TimeWeb** — 3–5 000 ₽/мес, у клиента

### Таблицы базы данных
| Таблица | Что хранит |
|---------|-----------|
| `telegram_messages` | Сообщения из чатов |
| `mymeet.meetings` | Транскрипты Zoom |
| `people` | Справочник сотрудников (222 чел. в пилоте) |
| `people_aliases` | Псевдонимы (247 в пилоте) |
| `tpi_scores` | Оценки TPI по неделям |
| `usage_log` | Расход токенов по клиентам |

## МОДЕЛЬ ВНЕДРЕНИЯ

### Ключевой принцип
```
Инфраструктура — у клиента
Мозги (методология, промпты, фреймворки) — у нас
```

### Техническая привязка (двойной замок)

**Замок 1 — GitHub репо (методология)**
SOUL.md, промпты, фреймворки — наше IP. Хранится в приватном репо.
При каждом рестарте сервера: `git pull` → агент получает актуальные инструкции.
При отключении клиента — убираем доступ к репо.

**Замок 2 — API ключ Claude**
Зашит в переменную окружения на сервере клиента.
Можно отозвать мгновенно → боты перестают работать через минуты.

**Замок 3 — Лицензионная проверка**
При рестарте Gateway — запрос на `api.systematika.ru/license`.
При `status: inactive` — агент блокируется независимо от API ключа.

### Что остаётся у клиента при уходе
✅ Сервер с данными, OpenClaw (open source), база данных
❌ Актуальная методология, API ключ Claude, веб-офис, поддержка

**Итог:** система физически стоит, но не работает без нашей IP.

### Процесс тиражирования
3 действия от клиента:
1. Зарегистрировать VPS на TimeWeb + API ключ
2. Создать два бота в BotFather + токены
3. Нажать «Запустить систему»

Что происходит автоматически (15 минут): создание VPS → Docker → OpenClaw + PostgreSQL → клонирование systematika-core → запуск ботов → уведомление клиента.

## ВЕБ-ОФИС КЛИЕНТА

**Концепция:** персональный веб-интерфейс на `office.systematika.ru/client_name`.

**Принцип:** агенты появляются в офисе только после подключения ядра.
Нет данных → нет агентов → нет офиса.

### Главная страница
- Дашборд Team Power Index: рейтинг + динамика
- Активность за 24 часа
- Экстренные сигналы
- Задачи на сегодня

### Карта команды
Каждый сотрудник — карточка:
- TPI балл (0–10)
- Динамика: ↑ растёт / → стабильно / ↓ падает
- Цвет: 🟢 норма / 🟡 внимание / 🔴 риск
- Клик → детальный разбор + график + рекомендации

### Логика появления агентов
```
Подключили чаты → Аналитик коммуникации
Подключили зумы → анализ встреч
Подключили CRM → Аналитик продаж
Подключили таскер → Контроль задач
Всё подключено → полный доступ + Стратег
```

### Технически
- Веб-приложение (HTML/JS), хостится у Систематики
- Подключается к тому же Gateway что и Telegram-бот
- Авторизация через Telegram или email
- Уровни доступа: CEO (всё), руководитель отдела (свой отдел), HR (метрики без переписок)

## GROWTH ENGINE

**Принцип:** каждый копайлот видит данные и автоматически предлагает следующий шаг.

### Архитектура
```
Данные клиента → [Signal Detector] → [Trigger Engine] → [Action Selector] → [Delivery Agent] → Персональное предложение
```

### Ключевые сигналы и действия
| Сигнал | Действие |
|--------|---------|
| TPI продавцов падает 2 недели | Предложить Sales Copilot |
| В зумах часто говорят о воронке | Рекомендовать Sales Copilot |

Не реклама — инсайт из данных:

Принцип «активировать, а не спрашивать»: первый отчёт делается автоматически, потом доставляется результат. Клиент уже видит ценность — нечего отклонять.

## МАППИНГ СОТРУДНИКОВ

**Проблема:** один человек в разных системах называется по-разному.
Telegram: @romabrmv / Zoom: «Роман» / CRM: «Роман Абрамов»

**Решение:**
- Таблица `people` — canonical_name, telegram, email, отдел, руководитель
- Таблица `people_aliases` — псевдонимы + источник + confidence score
- Скрипт `match_people.py` — нечёткое совпадение (fuzzy matching)

**Пороги confidence:**
- ≥ 0.9 → добавляем alias автоматически
- 0.8–0.89 → требует ручной проверки
- < 0.8 → требует ручного добавления

**Статус пилота:** 222 человека, 247 псевдонимов. Проблема: поле `telegram` пустое — нет маппинга с чатами (критический пробел).

## ОНБОРДИНГ КЛИЕНТА (7–14 дней)

**Этап 0 — Квалификация**
- Команда 5–100 чел
- Рабочие чаты (Telegram/Bitrix/VK)
- Онлайн-встречи (Zoom/Meet)
- CRM или таскер
- Какая-то отчётность сотрудников

**Этап 1 — Распаковка (1–2 часа)**
Аудит коммуникации + данных + внедрение регламентов.

**Регламенты для клиента:**
1. Вся рабочая коммуникация — только в чатах (не в личке)
2. Запись всех рабочих встреч (автозапись в Zoom)
3. Ежедневный отчёт сотрудника по шаблону (к 18:00)
4. CRM и таскер: все задачи и сделки только там
5. Справочник соответствий: кто есть кто в каждой системе

**Шаблон ежедневного отчёта сотрудника:**
```
МЕТРИКИ (план/факт):
• [Метрика]: план — X, факт — Y

САМОАНАЛИЗ:
Что получилось и почему? (2-3 предложения)

ТОЧКИ РОСТА:
Что мог сделать лучше?

ВОЗРАЖЕНИЯ / СЛОЖНЫЕ СИТУАЦИИ:
Какую сложность встретил? Как отреагировал?
```

**Этапы 2–4:** технический сетап → тестовая неделя → запуск и передача.

## РИСКИ

**Блокировка Telegram:**
- Сценарий 1 (частичная): работает через прокси (Telethon поддерживает SOCKS5/MTProto). Влияние минимальное.
- Сценарий 2 (жёсткая + массовый уход): критично. Митигация: архитектура коннекторов заложена, новый мессенджер добавляется без переписывания системы.

**Конкуренция с BigTech:**
Anthropic/OpenAI/Microsoft идут в упрощение и mass-market. Но им не интересны «мелкие рынки» — им проще зарабатывать на транзакциях. Sistematika = кастомизированные настройки поверх платформ, которые BigTech сам же и строит.

### База знаний — архитектура на примере реального внедрения

**Формат хранения (рекомендации из практики):**

- Каждый лист таблицы — отдельный файл

**Важно:** Claude предпочтительнее GPT для базы знаний с большим объёмом данных.

### Три применения базы знаний в продажах

1. **Агент продаж** — анализирует звонки, находит слабые места скриптов, повышает конверсию
2. **Агент клиентского сервиса** — анализирует переписки, выявляет типичные возражения
3. **Аналитик отчётов** — summary метрик по отделам, дайджест за день

### Чекер видео / Модерационный инструмент

> Разработка Кирилла для Content Hunter

**Функции:**
- Загрузка 15 видео клиента
