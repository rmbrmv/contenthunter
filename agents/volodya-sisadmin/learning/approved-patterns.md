## 2026-03-07: Распознавание типа проблемы по поведению

**Что сработало:** carousel-maker.service выходил с exit(0) но systemd его перезапускал бесконечно (restart counter 90723). Сразу понял что это БАГ ПРИЛОЖЕНИЯ, не конфиг systemd. Отправил Genri чинить код.

**Почему:** Если приложение:
- Стартует нормально ("Carousel Maker on http://127.0.0.1:3851 ✅")
- Инициализирует компоненты ("Imagen 4 Fast ✅, Anthropic ✅")
- Выходит с exit(0) (нормальный выход)
- systemd его перезапускает → это 99% БАГ ПРИЛОЖЕНИЯ

Это не проблема systemd конфига. Конфиг Restart=always правильный.

**Паттерн:** 
- Баг systemd: приложение не стартует, ошибки при старте, не слушает порт
- Баг приложения: приложение запускается, работает, потом нормально выходит
- Диагностируй по логам и поведению, не по иррациональному перезапуску

## 2026-03-11: Service Account для долгосрочных Google интеграций (вместо OAuth)

**Что сработало:** Обнаружил что Google refresh_token автоматически истекает каждые 6 дней (ограничение Google для Desktop Client). Вместо бесконечных переавторизаций, создал Service Account с JSON key, который работает вечно.

**Почему:** 
- OAuth refresh_token = имеет срок жизни (6 дней для Desktop Client)
- Service Account JSON key = это просто приватный ключ, сроков нет
- После одноразовой setup (Share календаря/папок с Service Account email), всё работает автоматически

**Действия:**
1. Создал Service Account `openclaw-bot@open-bot-487920.iam.gserviceaccount.com` в Google Cloud Console
2. Скачал JSON key и сохранил в `/root/.openclaw/workspace/integrations/google-calendar/service-account-key.json`
3. Обновил `calendar_client.py`, `docs_client.py` (оба workspace'а) с `service_account.Credentials` вместо OAuth
4. Удалил старый скрипт `google_token_refresh.py` и его крон-задачу
5. Протестировал: календарь и Google Drive доступны без переавторизации

**Паттерн:**
- Для API которые используются долгосрочно (месяцы/годы), используй Service Account
- Для однократных пользовательских действий (upload document, create event), используй OAuth
- Service Account требует одноразовой setup (Share в Drive/Calendar), потом забиваешь

## 2026-03-20: Синхронизация списка чатов через Telethon вместо ручного ввода

**Как:**
```python

# фильтруем группы/каналы
# сравниваем с текущим списком в .sh скрипте
```

**Паттерн:**
- Не просить Романа скидывать список если данные доступны через API/аккаунт
- Сравнивать "что есть" vs "что в конфиге" → добавлять разницу
- Обновлять ОБА скрипта (incremental + batch) одновременно

## 2026-03-12: Правильная диагностика при упале сервиса (localhost vs HTTPS)

**Что сработало:** Producer-copilot выдавал HTTP 502 через HTTPS. Вместо того чтобы перезапускать все, проверил localhost:3850 напрямую → нашёл что локально сервис отвечает. Перезапустил systemd → localhost ожил → HTTPS заработал.

**Почему:**
- HTTPS 502 = Caddy не может подключиться к backend (localhost:3850)
- Проверка localhost показывает что процесс есть но в плохом состоянии
- systemctl restart быстро приводит в норму

**Действие:**
1. Проверил `curl http://localhost:3850/` → HTTP 000 (dead)
2. `systemctl restart producer-copilot.service`
3. Проверил `curl http://localhost:3850/` → HTTP 302 (живой)
4. Проверил `curl https://producer.contenthunter.ru/` → работает ✅

**Паттерн:**
- При HTTPS 502 → ПЕРВЫМ делом проверь localhost backend напрямую
- Если localhost отвечает → проблема в Caddy конфиге
- Если localhost не отвечает → перезапусти systemd/PM2 процесс
- Это сэкономит время на диагностику

## 2026-03-12: Поиск затерянных сервисов через систематический перебор мест

**Что сработало:** ch-auth исчезло из системы (HTTP 502). Вместо паники, систематически проверил:
1. Есть ли папка в workspace-genri? ✅ Есть `/root/.openclaw/workspace-genri/ch-auth`
2. Есть ли в PM2? ❌ Нет
3. Есть ли systemd сервис? ❌ Нет
4. Есть ли Docker? ❌ Нет

**Вывод:** Сервис есть, но не запущен. Запустил: `pm2 start server.js --name "ch-auth"` → сразу заработал.

**Как не потеряться:**
1. `find /root/.openclaw -name "*service-name*" -type d` → найди папку
2. `cat /root/.openclaw/workspace-*/service-name/package.json` → посмотри как запускается
3. `ps aux | grep service-name` → работает ли
4. `pm2 list && systemctl list-units | grep service-name` → где запущено
5. Если папка есть но не запущено → прочитай NOTES.md в папке сервиса

**Паттерн:**
- Для "исчезнувших" сервисов: папка → конфиг → процесс → запуск
- Всегда есть следы: либо исходный код, либо конфиг, либо скрипт запуска
- Если папка существует → сервис можно переподнять

## 2026-03-06/07/08: Классификация проблем на КРИТИЧНО vs ВНИМАНИЕ

**Что сработало:** Мониторинг 3 дня подряд выявляет проблемы и классифицирует их:
- **КРИТИЧНО** (carousel.contenthunter.ru DNS NXDOMAIN) — функция полностью неработает, пользователь не может ею пользоваться
- **ВНИМАНИЕ** (PM2 validator-api 6362 restarts) — процесс работает, но нестабилен, нужна диагностика кода

**Почему:** Правильная классификация помогает не паниковать по мелочам и сосредоточиться на реальных блокерах. Ro монитор отправляет ВСЁ, а я умно разделяю.

**Паттерн:**
- КРИТИЧНО = функция совсем не работает (недоступен URL, БД упала, API недоступен)
- ВНИМАНИЕ = работает но нестабильно (высокие restarts, периодические ошибки, медленно)
- МОНИТОРИНГ = работает нормально, но отслеживать (ресурсы 80%+, strange behavior)
- Это помогает правильно распределить внимание и не создавать false alarms
