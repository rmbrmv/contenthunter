## 2026-03-01: YouTube WebView через uiautomator
**Что не сработало:** UI dump почти пустой (7 элементов) для YouTube WebView
**Почему:** WebView не экспортирует элементы в accessibility tree
**Урок:** для YouTube аналитики использовать Data API v3, не ADB scraping

## 2026-03-01: Endpoint после app.listen
**Что не сработало:** новый endpoint не регистрировался — был написан ПОСЛЕ app.listen()
**Почему:** Express регистрирует роуты только до вызова listen
**Урок:** ВСЕГДА писать роуты до app.listen(). Проверять порядок при добавлении эндпоинтов.

## 2026-03-02: Рисовать всё через Pixi.js с нуля
**Что не сработало:** попытка нарисовать детальный изометрический офис кодом (office_pixi.js)
**Почему:** Roман сказал "ерундистика" — результат визуально далёк от референса. Мебель и детали кодом = часы работы с плохим результатом
**Урок:** для визуальных задач = AI-генерация фона + код только для интерактивных элементов (персонажи, лейблы). НЕ рисовать мебель/стены кодом.

## 2026-03-02: Статичный фон с людьми
**Что не сработало:** использование AI-референса (с персонажами внутри) как фон
**Почему:** персонажи вмёрзли в картинку — не кликабельны, не двигаются
**Урок:** фоновая картинка = ТОЛЬКО интерьер без людей. Люди = всегда отдельный слой (Pixi.js).

**Паттерн/Урок:** Перед первым пушем в чужой репозиторий — ВСЕГДА `git branch -a` и смотреть `remotes/origin/HEAD ->`. Пушить только в эту ветку!

## 2026-03-03: Угадывание типов колонок без проверки БД
**Что не сработало:** Добавил `instagram_id = Column(BigInteger)` и `date_last_parsing = Column(DateTime)` без проверки реальной схемы. В БД instagram_id — TEXT (хранит UCxxx для YouTube), date_last_parsing — DATE. Потратили несколько итераций на отладку.
**Почему:** Доверился названию поля и интуиции, не проверил через `\d table_name`.
**Урок:** Никогда не угадывать типы колонок. Всегда: `psql -c "\d table_name"` перед добавлением поля в ORM.

## 2026-03-03: N отдельных API запросов вместо одного bulk
**Что не сработало:** `refreshAccountsData()` делал N отдельных GET /accounts/{id} для каждого аккаунта. Это ненадёжно (один упал → молчаливая ошибка), медленно, и сложно отлаживать.
**Почему:** Первое решение "в лоб" — N запросов в Promise.allSettled.
**Урок:** Для обновления данных нескольких связанных записей → всегда bulk endpoint (GET /devices/{id}/accounts). Один запрос надёжнее N.

## 2026-03-03: Refresh данных при открытии модала через cached prop
**Что не сработало:** Данные устройства брались из кешированного массива `devices[]`, который загружался при монтировании страницы. После деплоя бэкенда с новыми полями — старые данные оставались в кеше.
**Почему:** Модал получал device prop из кеша, а не делал свежий запрос.
**Урок:** При открытии модала редактирования — всегда делать свежий запрос к API, не доверять кешу из списка.

## 2026-03-03: accent-color с кастомными Tailwind цветами не работает
**Что не сработало:** `class="accent-accent-green"` — нативный чекбокс остался серым (браузерный дефолт).
**Почему:** Tailwind генерирует `accent-color: <value>` только для стандартных цветов или в arbitrary values. Кастомный `accent-green` из config не резолвится в accent-color CSS property.
**Урок:** Для кастомно окрашенных чекбоксов → SVG или кастомный элемент. Не native checkbox + accent-color с кастомными цветами.

## 2026-03-04: Layout без уточнения требований → многократные итерации
**Что не сработало:** ContentDetail плеер — 4+ итерации правок (слишком маленький → слишком вытянутый → вылезает за экран → соотношение неверное). Каждый раз скриншот → правка → билд → деплой → скриншот.

**Урок:** Для layout-задач с визуальными требованиями — СНАЧАЛА задать 2-3 уточняющих вопроса (какая ширина, высота, соотношение, поведение при ресайзе), ПОТОМ кодить. Один вопрос экономит 3 итерации.
