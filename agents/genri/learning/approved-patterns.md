## 2026-03-03: Паттерн таблиц для validator.contenthunter.ru

**Что сработало:** Единый паттерн для всех таблиц/журналов/справочников в проекте validator (и шире — во всех проектах Content Hunter).

**Обязательные элементы каждой таблицы:**
1. **Строка поиска** — над таблицей, `max-w-md`, поиск по основным полям
2. **Заголовки с сортировкой** — клик 1: ↑ asc, клик 2: ↓ desc, клик 3: ⇅ off; класс `th-sort`; компонент `SortIcon`
3. **Строка фильтров** — вторая строка thead, `bg-indigo-50/50`; текстовые инпуты или `<select>` из реальных значений; кнопка "Сбросить" если хоть один фильтр активен
4. **Счётчик** — `Показано: X из Y` в футере таблицы
5. **Кнопка добавить** — вверху справа, `bg-indigo-600`, `＋ Добавить ...`
6. **Колонка Действия** — ✏️ (редактировать, tooltip) и 🗑️ (удалить, tooltip)
7. **Модал редактирования** — предзаполненные поля + Сохранить/Отмена
8. **Модал удаления** — «Вы подтверждаете удаление X?» + Да, удалить/Отмена (красная кнопка)

**CSS классы (scoped):**
- `.th-sort` — заголовок с сортировкой (hover, cursor-pointer)
- `.filter-input` — фильтр в строке thead
- `.input-field` — поле в модале
- `.field-label` — лейбл поля в модале

**Логика (computed `processed`):** фильтрация → сортировка → отдаём в `v-for`

**Паттерн:** использовать во ВСЕХ таблицах проекта validator и других сервисов Content Hunter.

## 2026-03-03: SVG вместо native checkbox для кастомных стилей
**Что сработало:** Заменил `<input type="checkbox" accent-accent-green>` на кастомный SVG — ярко-зелёный активный, серый неактивный.
**Почему:** `accent-color` с кастомными Tailwind цветами (`accent-green`) не работает в браузере — цвет не резолвится.
**Паттерн:** Для кастомно стилизованных чекбоксов/переключателей — всегда SVG или span+CSS, не native input с accent-color.

## 2026-03-04: Диагностика API через curl перед правкой кода
**Что сработало:** Вместо угадывания почему поле не отображается — сразу `curl /api/content/6 | python3 -c "print(d.keys())"` → увидел что mime_type отсутствует в ответе → точечный фикс в роутере.
**Почему:** Прямой вызов API даёт точную картину что возвращает бэкенд, без догадок.
**Паттерн:** При несоответствии данных на фронте — СНАЧАЛА проверить API curl/httpie, ПОТОМ лезть в код.

## 2026-03-04: Проверка статической директории сервера перед копированием ассетов
**Что сработало:** `grep -n "static\|sendFile\|public" server.js` — сразу видно что сервер раздаёт из `public/`, а не из корня.
**Почему:** Node/Express может раздавать из любой директории; угадывать = терять время.
**Паттерн:** Перед копированием ассетов в новый сервис — проверить `express.static(...)` в server.js. Ассеты класть только туда куда указывает static middleware.

## 2026-03-06: Разблокировка экрана ADB перед каждой публикацией
**Что сработало:** `KEYCODE_WAKEUP` + `wm dismiss-keyguard` + проверка `screenState` — обязательный шаг перед любым ADB UI действием.
**Почему:** Samsung гасит экран каждые 30 сек (mLockScreenTimeout=30000). Без пробуждения — intents уходят в lockscreen, UI dump пустой, публикация "молча" не выполняется.
**Паттерн:** `ensure_unlocked()` всегда первым шагом в publisher.py + `svc power stayon true` + `screen_off_timeout 600000` на старте задачи.

## 2026-03-06: ACTION_SEND без -p для TikTok (системный chooser)
**Что сработало:** `am start -a android.intent.action.SEND -t "video/mp4" --eu android.intent.extra.STREAM "content://..."` БЕЗ `-p` → системный chooser → TikTok → видео-редактор.
**Почему:** `-p com.zhiliaoapp.musically` открывает TikTok на главной, не в редакторе загрузки. Chooser направляет в нужный activity.
**Паттерн:** Для видео-шаринга в соцсети через ADB: ACTION_SEND + content URI + WITHOUT -p. Пусть системный chooser выбирает target.

## 2026-03-06: content:// URI вместо file:// для Android 10+
**Что сработало:** MediaStore query → `content://media/external/video/media/{id}` → передаётся в intents вместо `file:///sdcard/...`.
**Почему:** `file://` URI заблокирован в Android 10+ (FileUriExposedException). Приложения игнорируют такие intents.
**Паттерн:** Перед шарингом файла через ADB intent — зарегистрировать в MediaStore → получить `content://` URI. Метод: `_get_mediastore_content_uri(remote_path)`.

## 2026-03-06: Контекстная справка HELP_CONTENT — встроенная документация
**Что сработало:** JS-объект `HELP_CONTENT` с Markdown прямо в index.html + `helpSetSection()` при каждом `nav()` вызове → панель всегда актуальна при переходах между разделами.
**Почему:** Документация рядом с кодом → не теряется, обновляется вместе с фичей в одном коммите.
**Паттерн:** При добавлении нового раздела — сразу добавить `HELP_CONTENT['section'] = 'markdown...'`. Правило: фича без справки = незаконченная фича.

## 2026-03-06: processing → cron автозаполнение post_url
**Что сработало:** Статус `processing` (не `done`) после шаринга + отдельный cron каждые 10 мин скрейпит профиль в соцсети → когда URL появился → `done`. Ручное одобрение убрано.
**Почему:** Соцсети обрабатывают видео 30–90 сек. Ждать синхронно нельзя — блокирует publisher. Асинхронный cron = правильная архитектура.
**Паттерн:** Для операций с неизвестным временем завершения (загрузка видео, обработка) → промежуточный статус + polling cron. Не ждать синхронно.

## 2026-03-05: E2E тест пайплайна (30 схем, 0 ошибок)
**Что сработало:** После всех фиксов — запустить задачу с 30 схемами и убедиться что все прошли (done, 30/30 OK). Не считать работой только факт что "процесс не упал".
**Почему:** Реальный тест с большим числом схем выявил что fix работает стабильно, не только в одном случае.
**Паттерн:** После критических фиксов worker/cron — обязательно прогонять нагрузочный тест (N схем, не 1).

## 2026-03-05: pack_scheme_map в meta как позиционный маппинг
**Что сработало:** Хранить `{"scheme_id": pack_id}` в `unic_tasks.meta` при создании задачи. При назначении результатов в publish_queue — достать из meta, не вычислять заново.
**Почему:** Паки/схемы могут измениться между созданием задачи и её выполнением. Фиксация маппинга в момент создания = детерминированный результат.
**Паттерн:** Для любого pipeline-шага где нужен маппинг A→B — фиксировать его в meta при старте, не перевычислять на каждом шаге.

## 2026-03-03: sticky th вместо sticky tr/thead
**Что сработало:** Перенёс `sticky top-0` с `<tr>` на каждый `<th>` — заголовки начали прилипать.
**Почему:** В HTML таблицах `position: sticky` работает только на `<th>` или `<td>`, не на `<tr>` или `<thead>`.
**Паттерн:** Sticky в таблице → ВСЕГДА на каждом `<th>` отдельно, с явным `bg-*` фоном на каждом.

## 2026-03-03: overflow-hidden на предке ломает sticky
**Что сработало:** Убрал `overflow-hidden` с MainLayout `<main>` и с outer wrapper DevicesPage — sticky заработал.
**Почему:** Любой `overflow: hidden/auto/scroll` на предке между sticky-элементом и scrolling-контейнером отключает sticky.
**Паттерн:** Если sticky не работает — искать `overflow-hidden` у всех предков. Scrolling контейнер должен быть прямым скроллящим родителем sticky-элемента.

## 2026-03-03: Глобальный тёмный скроллбар через ::-webkit-scrollbar
**Что сработало:** Один блок в style.css покрыл все скроллбары во всём приложении (модал + таблица + sidebar).
**Почему:** ::-webkit-scrollbar применяется глобально + Firefox через scrollbar-width/color.
**Паттерн:** Тёмная тема → добавить в style.css: ::-webkit-scrollbar 6px, track #1a1e2e, thumb #2d3348. Для Firefox: * { scrollbar-width: thin; scrollbar-color: thumb track }.

## 2026-03-11: dump_ui() — чистка XML-префикса обязательна
**Что сработало:** `_clean_xml(s)` стрипает всё до `<?xml` перед возвратом. ET.fromstring начал корректно парсить, clickable-элементы стали видны в диагностике редактора Instagram.
**Почему:** UIAutomator пишет «UI hierchary dumped to: /sdcard/ui.xml» перед XML — это часть stdout, не заголовок XML.
**Паттерн:** В `dump_ui()`: `idx = s.find('<?xml'); return s[idx:] if idx > 0 else s`. Всегда. В любом новом сервисе где используется UIAutomator dump через adb shell.

## 2026-03-11: Пошаговое логирование в publisher.py + log_event на каждом шаге
**Что сработало:** После добавления `log_event('info', 'Instagram: шаг 5 — выбор видео из галереи')` на каждом шаге — за один прогон нашли точный шаг сбоя (редактор, шаг 6, неверные координаты). До этого 8 задач падало без понятной причины.
**Почему:** Подробные события видны в UI-модале «Лог событий» в реальном времени. Без них — только статус «failed».
**Паттерн:** Для каждого нового UI-флоу в publisher.py: `log_event('info', 'Шаг N — описание')` в начале шага + `log_event('error', ...)` при падении. Минимум 5-7 событий на флоу.

## 2026-03-11: Поиск кнопки в XML по resource-id а не только по тексту
**Что сработало:** При поиске кнопки «Далее» в Instagram Reels редакторе — добавили поиск по `resource-id` паттернам (`next`, `forward`, `continue`) + по bounds в правой части экрана (x>800, y<300). Это надёжнее чем угадывать текст кнопки.
**Почему:** Кнопки с иконками (стрелки, шевроны) не имеют text/content-desc. resource-id стабильнее координат.
**Паттерн:** Для кнопок без текста: 1) resource-id с 'next'/'forward'/'continue' 2) clickable ImageButton в ожидаемой зоне экрана. Не хардкодить координаты.

## 2026-03-11: Строгие маркеры экрана вместо ненадёжных ключевых слов
**Что сработало:** Вместо `if 'Поделиться' in ui:` (ложные срабатывания от навбара) → `STILL_ON_DESC = ['Добавьте описание', 'Приватность', 'Черновики']`. Плюс проверка `clickable=true` для «Поделиться».
**Почему:** Одно слово встречается в разных контекстах (кнопка публикации vs навбар vs другой экран). Маркер должен быть уникальным для конкретного экрана.
**Паттерн:** Для каждого экрана в publisher.py — определить 2-3 УНИКАЛЬНЫХ маркера. Не использовать слова которые встречаются в навбаре/статус-баре. Проверять по UI dump.

## 2026-03-11: pm2 restart как единственный выход из кэш-дедлока
**Что сработало:** Когда scheduler кешировал устаревшие настройки (loadSettings после isWorkingHours) — pm2 restart вызывал init() → loadSettings с актуальными данными.
**Почему:** `init()` вызывает loadSettings() до первого tick(), поэтому рестарт всегда сбрасывает кэш.
**Паттерн:** Если меняешь critical settings в autowarm_settings через psql — делай `pm2 restart autowarm`. Не ждать автообновления кэша — в scheduler.js loadSettings ПОСЛЕ working hours check.

## 2026-03-12: Ручное наблюдение как единственный надёжный источник флоу
**Что сработало:** Observer script (PM2 + UIAutomator dump каждые 2 сек) во время ручной публикации → точные маркеры экранов, activity names, UI тексты в реальном времени. За 3 ручных прогона (YouTube/TikTok/Instagram) исправили все ключевые баги флоу.
**Почему:** Автоматические прогоны дают только "упало/не упало". Observer даёт точную картину что на экране в каждый момент — без догадок.
**Паттерн:** При любом новом флоу публикации или при повторяющихся падениях — запускать `obs2.py` через PM2, делать ручной прогон, читать лог. Не угадывать маркеры "на глаз".

## 2026-03-12: Observer через PM2 — единственный способ выживать при Gateway рестартах
**Что сработало:** `pm2 start obs2.py --interpreter python3` вместо `exec(background=True)` — скрипт не убивается при рестарте OpenClaw Gateway.
**Почему:** exec-сессии живут в процессе Gateway и умирают вместе с ним. PM2 = отдельный process manager, изолирован от Gateway cgroup.
**Паттерн:** Любой фоновый скрипт который должен жить дольше одного запроса → запускать через PM2. После завершения — `pm2 delete <name>`.

## 2026-03-12: topResumedActivity как индикатор overlay
**Что сработало:** `dumpsys activity activities | grep topResumedActivity` в confirmation loop — определяет КАКОЙ package на переднем плане. Samsung AddItemActivity (`com.sec.android.app.launcher/.AddItemActivity`) был причиной всех TikTok падений.
**Почему:** overlay_streak по activity = единственный надёжный способ детектировать что появился системный диалог поверх приложения.
**Паттерн:** В любом confirmation loop для публикации — всегда проверять topResumedActivity, не только UI текст. Samsung добавляет неожиданные диалоги (виджеты, права, оценки) которые UIAutomator видит с задержкой.

## 2026-03-12: SUCCESS detection через activity transition, не только UI текст
**Что сработало:** После публикации Instagram → MainTabActivity (немедленно). После TikTok → SplashActivity + профиль / DetailActivity. YouTube → InternalMainActivity + "Ваши видео". Activity transition = надёжнее чем "Reel опубликован" (которого нет).
**Почему:** Тексты success-экранов угадывались неверно. Activity name стабилен и не локализован.
**Паттерн:** При добавлении SUCCESS detection → проверить через наблюдение КАКАЯ activity появляется после публикации. Это первичный индикатор. UI текст — вторичный.

## 2026-03-19: Блокировка действия с redirect к зависимому условию
**Что сработало:** При клике на слот планировщика — сначала `GET /api/schemes/summary/{project_id}`. Если `approved < min_required` → модалка-блокировка с кнопкой «Перейти к одобрению схем» → `/client/schemes`. Не просто ошибка, а конкретный путь к решению.
**Почему:** Пользователь понимает что мешает и куда идти — не гадает. UX-паттерн "constructive error".

## 2026-03-19: Round-robin ротация с offset от последней задачи
