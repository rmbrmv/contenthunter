## 2026-03-23 — Фикс хромокея в превью схем уникализации

### Проблема
На client.contenthunter.ru (/client/schemes) в превью схем уникализации хромокей (#00ff30) не вырезался — оверлей-видео показывалось с зелёным фоном.

### Причина
В таблице `validator_unic_content` поле `chromakey_color` было `0x19af3e` (тёмно-зелёный, неверный).
FFmpeg-фильтр `chromakey=0x19af3e:...` не попадал по реальному цвету видео `#00ff30` → фон не вырезался.

### Исправление
```sql
UPDATE validator_unic_content 
SET chromakey_color = '0x00ff30' 
WHERE content_type = 'video' AND chromakey_color = '0x19af3e';
-- Обновлено 10 записей
```

Перегенерированы превью для Тестового проекта (id=10), 30/30 схем — хромокей вырезается корректно.

### Правило
- Цвет хромокея всех оверлей-видео: `0x00ff30` (= #00ff30)
- При добавлении новых видео в `validator_unic_content` — проверять это поле
- FFmpeg: `chromakey=0x00ff30:0.12:0.02` + `format=rgba`

## validator — async background validation при загрузке (2026-03-23, Генри)

**Проблема:** `POST /api/upload/file` падал с 502 — валидация (транскрипция Groq Whisper + OCR + модерация) занимала 60–120 сек, Caddy таймаутил.

**Решение (Вариант B — async):**

### Бэкенд (upload.py)
- `POST /api/upload/file` теперь возвращает ответ **сразу** после загрузки в S3 и сохранения записи в БД (`status=validating`)
- `_do_full_validation` запускается в `asyncio.create_task` с **отдельной сессией БД** через `AsyncSessionLocal` (важно: не переиспользовать сессию запроса!)
- Новый endpoint: `GET /api/upload/status/{content_id}` — опрос статуса, возвращает `{id, status, is_done, validation}` где `is_done=true` когда статус не `validating`

### Фронт (UploadModal.vue)
- После загрузки при `status=validating` — запускает polling каждые 3 сек на `/api/upload/status/{id}`
- Показывает анимированные стадии: «Транскрибируем аудио...» → «Распознаём текст на кадрах...» → «Проверяем модерацию...» → «Оцениваем виральность...» → «Почти готово...»
- Таймаут 5 минут → ошибка пользователю
- После `is_done=true` → отображает результат (UI без изменений)

**Правило:** при любом длинном фоновом процессе на бэке — `asyncio.create_task` с новой `async with AsyncSessionLocal() as bg_db` — никогда не переиспользовать сессию из request context.

**Коммит:** `8095ec8` в GenGo2/validator-contenthunter

## autowarm — Фикс навигации: сайдбары не кликались (2026-03-23)

### Симптомы
- Все пункты боковых меню (сайдбаров) не работали
- Кнопка справки `?` не работала
- Верхние табы (`switchModule`) работали нормально

### Причина 1 — JS SyntaxError в HELP_CONTENT (главная)
В тексте справки `public/index.html` были backtick-символы внутри JS template literal:
- `` `https://save.gengo.io/...` `` — коммит `5fb596c`/`f7b0ba3`
- `` `/sdcard/debug_screenshots/` `` — тот же блок документации

SyntaxError → функция `helpSetSection()` не определялась → `nav()` (вызывает `helpSetSection` первой строкой) падал с ReferenceError → сайдбары молчали.

**Фиксы:** `4911b96`, `6f7066c` — убраны backtick вокруг URL и путей в HELP_CONTENT.

### Причина 2 — global-settings не в списках скрытия сайдбаров
Коммит `85ea345` добавил вкладку Настройки, обновил список только в `switchModule()`. В `nav()` и `openEditor()` остались старые списки — `sidebar-global-settings` не скрывался при переходах.

**Фикс:** `eb134bd` — `'global-settings'` добавлен в оба списка.

### Причина 3 — ETag кэширование
`express.static` отдавал ETag → браузер получал 304 → получал старый JS. Исправлено: `etag: false, lastModified: false`.

**Фикс:** `dacf92f`

### Правило для разработчиков
> В HELP_CONTENT (JS template literal `\`...\``) нельзя использовать backtick внутри текста.
> Вместо `` `path/to/file` `` писать просто `path/to/file` или экранировать: `\\\`path\\\``.
