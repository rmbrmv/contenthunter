## autowarm — фикс счётчика лайков фарминга (2026-03-25)

**Проблема (задача #62):** лайки ставились (10 штук в Фазе 0 — поиск), но в UI и БД показывалось `likes=0`.

**Причина:** `watch_content()` (Фаза 2) писал `progress.likes = likes_done` где `likes_done=0` (в ленте была только реклама) → перезаписывал `search_likes=10` нулём.

**Фикс (коммит `ac129be`):**
- `run()`: инициализирует `self._search_likes_today = 0`
- Фаза 0: `self._search_likes_today = search_likes` после `run_search_phase()`
- Основной цикл: `_update_progress({'likes': likes_done + self._search_likes_today, ...})`
- Summary Фазы 2: `total_likes = likes_done + search_likes` + разбивка в логе `(лента=N, поиск=M)`
- `likes_done` в `return` теперь возвращает `total_likes`

**Правило:** `progress.likes` = всегда СУММАРНЫЕ лайки (лента + поиск). `progress.search_likes` = только поиск (для аналитики). В UI смотреть `progress.likes` — там уже итог.

## autowarm — 4 баг-фикса фарминга (2026-03-25, задачи #99/#100/#101)

**Коммит:** `7d145f5` → GenGo2/delivery-contenthunter

### Фиксы:
1. **detect_ad** — паттерн больше не ищет «реклам» в весь XML, только в `text=`/`content-desc=` атрибутах. Счётчик `_consecutive_ads`: если >4 подряд → сброс + warning (ложная серия или пустой фид).

2. **like_content** — добавлен `_verify_like()`: после тапа берём UI dump и проверяем что лайк отобразился (Instagram: `Unlike`, TikTok: `selected=true`/`liked`, YouTube: `selected=true`). Без подтверждения лайк не засчитывается.

3. **verify_and_switch_account** — retry 3×5с вместо тихого `continue`. Если все 3 попытки упали → `return False` → задача останавливается с `failed`.

4. **scheduler.js watchdog** — `watchdogStuckTasks()` в каждом `tick()`: если задача в `running` > 2ч без обновления `updated_at` → принудительно `failed`.

### Симптомы которые исправлены:
- Лайки в логах есть, но тестировщик говорит их не было → фикс #2
- 11 подряд «Реклама — пропуск» при нормальном контенте → фикс #1

## autowarm — универсальное сохранение суб-навигации в URL (2026-03-25)

**Задача:** при F5 в любом подразделе сервиса delivery.contenthunter.ru пользователь оставался в том же подразделе.

**Решение:** универсальный механизм `?sub=key:value` в URL-хэше.

**Формат URL:** `#module/section?sub=key:value`
| Раздел | URL |
|--------|-----|
| Выкладка → Опубликовано | `#publishing/publishing?sub=up:tasks` |
| Выкладка → Запланировано | `#publishing/publishing?sub=up:queue` |
| Instagram → Задачи | `#accounts/instagram?sub=ig:tasks` |
| WhatsApp → Аккаунты | `#accounts/whatsapp?sub=wa:accounts` |
| Admin → Доступ | `#admin/admin?sub=admin:access` |

**Функции:**
- `setSubParam(value)` — записывает `?sub=` в URL
- `getSubParam()` — читает `?sub=` из URL
- `restoreSubState(sec, sub)` — восстанавливает таб/блок по ключу
- Вызывается из `upSwitchTab()`, `_showSubBlocks()`, `navAdmin()` при каждом переключении

**Критический баг в первой попытке:** `nav()` вызывал `history.replaceState()` и затирал `?sub=`. Фикс: читать `savedSub = getSubParam()` **до** `nav()`, передавать напрямую.

**Коммиты:** `b6495cf`, `c5d9cfa`, `851cd18` → GenGo2/delivery-contenthunter

## autowarm — фикс Instagram аккаунт-свитчера и верификации лайков (2026-03-25, Даниил)

**Коммиты:** `978fa98`, `29ef1f5`, `4ae95dc`, `e1a1d6b`, `a43e9d9` → GenGo2/delivery-contenthunter

### Проблема
Задачи фарминга 102–110 падали: «Аккаунт не активен и не удалось переключиться».
На телефонах несколько паков аккаунтов, алгоритм должен переключаться — не переключался.

### Причины и фиксы

**1. `switch_instagram_account` — сломанные регулярки** (`978fa98`)
- Координаты кнопки профиля не совпадали между `get_current` (990,2140) и `switch` (1030,2270) → унифицированы на `(972, 2137)` (проверено на реальных устройствах)
- Паттерн `text="..."[^>]*resource-id="...username..."` никогда не матчился (порядок атрибутов UIAutomator обратный) → заменён на content-desc + fallback
- `content-desc` только EN: «Switch to ...» → добавлен RU: «Переключиться на ...»
- Добавлена верификация после тапа через `get_current_instagram_account`

**2. `get_current_instagram_account` — Y<200 не захватывал username** (`4ae95dc`)
- Реальный bounds username: `[335,96][713,254]` → y2=254, порог `y2<200` пропускал
- Исправлено: `y2 < 300`

**3. `switch_instagram_account` — формат content-desc с уведомлениями** (`a43e9d9`)
- Instagram показывает: `content-desc="slimvi_1, 7 уведомлений"` (без «Switch to»)
- Добавлен паттерн `cd_bare_matches` для `content-desc="username, ..."` + clickable="true"
- bounds-поиск тоже обновлён под этот формат

**4. `_verify_like` — SurfaceView блокировал лайки** (`e1a1d6b`)
- Instagram Reels рендерится через SurfaceView → `uiautomator dump` возвращает пустой XML (<86 байт)
- Код видел пустой дамп → `return False` → лайк не засчитывался (0 лайков на всех задачах)
- Исправлено: `len(ui) < 200` → `return True` (доверяем двойному тапу)
- Явный `False` только если UI полный и Unlike не найден

**5. Скринкаст теперь стартует до verify_and_switch_account** (`29ef1f5`)
- Раньше: запись начиналась после проверки аккаунта → при падении на этапе аккаунта видео не было
- Теперь: `start_screen_record()` вызывается сразу после `initialize()`
- При падении `stop_and_upload_screen_record(False)` вызывается явно перед `return`

**Правила:**
- Кнопка профиля Instagram: `content-desc="Профиль"`, центр `(972, 2137)`, bounds `[864,2070][1080,2205]`
- Username в профиле: bounds Y2 до ~260px → порог должен быть ≥300
- Switcher может показывать `"username, N уведомлений"` — всегда парсить первую часть до запятой
- `uiautomator dump` в Instagram Reels → пустой XML из-за SurfaceView → не считать ошибкой
