## 10:44–11:12 UTC — Фикс video_trim_start + ошибка задачи #3

### Проблема: обрезка начала видео
В `generate_ffmpeg()` строка:
```python
ts=scheme.get('video_trim_start')
if ts and float(ts)>0: cmd.extend(['-ss',str(float(ts))])
```
Ставила глобальный seek `-ss` ПЕРЕД всеми инпутами → FFmpeg пропускал первые N секунд исходника.
Схемы 1-5, 15-25 имеют `video_trim_start` до 5 секунд → видео начиналось с середины фразы.

### Фикс
- Убрали `-ss trim_start` из FFmpeg команды полностью
- `video_trim_start` в БД остаётся, но НЕ применяется к обрезке
- `validate_output()`: убрали `effective_dur = orig_dur - trim_start` → теперь `effective_dur = orig_dur`
- Контент сохраняется целиком, скорость меняется только через `setpts` (ускорение/замедление)

### Правило
**НИКОГДА не обрезать контент исходного видео.** Только изменение скорости через setpts.
`video_trim_start` — мёртвое поле, не использовать для FFmpeg seek.

## 14:57–15:04 UTC — Уточнение модели пак-схема

### КЛЮЧЕВАЯ БИЗНЕС-ЛОГИКА: 1 пак = 1 схема уникализации
- Пак аккаунтов = связка Instagram + TikTok + YouTube одного клиента
- 1 клиент → N паков → N схем уника → N уникальных видео
- Каждый пак получает своё уникальное видео (свою схему)
- Правило: в 1 паке не может быть 2 активных аккаунтов одной платформы (антибан)
- НО в реальных данных нарушения есть → берём первый активный аккаунт по id на платформу

### Маппинг схема → пак (позиционный)
```
pack_accounts (project_id=X, ORDER BY id):
  пак[0] → scheme_ids[0]
  пак[1] → scheme_ids[1]
  ...
```
Маппинг фиксируется в `unic_tasks.meta` при создании задачи:
```json
{"pack_scheme_map": {"scheme_id_1": "pack_id_38", ...}}
```

### Цепочка данных для выкладки
pack_accounts → device_numbers → serial + raspberry_number
raspberry_port (factory DB): raspberry_number → adb_port (15037-15108) + host IP
factory_inst_accounts → pack_id → username + platform

### Пример: проект Relisme (project_id=9)
- Пак 38 (Relisme_90, device 90, serial RFGYB07Y65V): instagram:relisme_art | tiktok:relisme_art | youtube:WowRelisme
- Пак 50 (Relisme_70, device 70): instagram:relisme_fit | pinterest:... | tiktok:relismeedit | youtube:RelisMeGirl
- raspberry_port: raspberry_number → adb порт (15037, 15017 и т.д.)

### Изменения в план реализации
- Этап 2 (unic_results → publish_queue): использовать pack_scheme_map из meta
- Для каждого unic_result.scheme_id → найти пак → взять аккаунты пака → создать publish_queue по 1 на платформу
- Итого задач выкладки на одно видео = N платформ в паке (обычно 3: inst+tiktok+yt)

## 15:04–15:07 UTC — Уточнения по аккаунтам и устройствам

### Статус аккаунта (active)
- Аккаунты банятся → active=false; новый аккаунт добавляется в тот же пак
- При создании publish_queue: WHERE active=true, первый на платформу
- При запуске задачи: ПЕРЕПРОВЕРЯТЬ active=true (аккаунт мог забаниться после постановки в очередь)
- Если забанился → status='skipped', notes='аккаунт неактивен'

### Несколько паков на одном устройстве
- 1 устройство может иметь N паков из разных проектов
- Проблема: одновременные scheduled_at на одно устройство
- Решение: при создании publish_queue добавлять интервал между задачами для одного device_serial
  scheduled_at = MAX(scheduled_at WHERE device_serial=X AND status IN pending/running) + publish_interval
- publish_interval → из autowarm_settings
