# Архитектура системы Elena-HR v2

> Ключевой принцип: Data Engine Pipeline — единый конвейер от сырых данных до управленческих рекомендаций

---

## Общая схема системы

```
┌─────────────────────────────────────────────────────────────────┐
│                      ИСТОЧНИКИ ДАННЫХ                           │
│  [Telegram-чаты]  [Zoom-записи]  [Отчёты]  [CRM/Таскер]  [Опросы] │
└──────────────────────────┬──────────────────────────────────────┘
                           │  сырые события
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│                       DATA ENGINE                               │
│  Сбор → Нормализация → Расчёт метрик → Граф коммуникаций       │
│                  + Feedback Loop (коррекции руководителя)       │
└──────────────────────────┬──────────────────────────────────────┘
                           │  JSON-пакет метрик + confidence scores
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│                    СЛОЙ МЕТРИК                                  │
│  TPI (0-10) | Reliability | Expertise | Sentiment | Activity    │
│  Граф коммуникаций | Командный индекс                           │
└──────────────────────────┬──────────────────────────────────────┘
                           │  метрики распределяются по агентам
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│                      СЛОЙ АГЕНТОВ                               │
│  [HR-директор] [Администратор] [Социолог] [Аналитик] [Коуч]    │
│                    (параллельная интерпретация)                 │
└──────────────────────────┬──────────────────────────────────────┘
                           │  структурированные выводы
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│                   РЕКОМЕНДАЦИИ                                  │
│  Дашборд недели | Алерты | Коучинг-план | Управленческие инсайты│
└─────────────────────────────────────────────────────────────────┘
```

---

## Описание каждого слоя

### Слой 1 — Источники данных

Пять типов источников, каждый с собственным коннектором:

| Источник | Формат | Частота обновления |
|----------|--------|--------------------|
| Корпоративные чаты | JSON (Telegram API, Bitrix webhook) | Real-time |
| Встречи / Zoom | Транскрипт (TXT/SRT) + мета | После завершения встречи |
| Ежедневные отчёты | Текст/форма | Раз в день |
| CRM / Таскер | JSON API | Раз в 4 часа |
| Пульс-опросы | JSON-форма | По расписанию |

**Правило:** источники не читаются агентами напрямую. Только Data Engine имеет доступ к сырым данным.

---

### Слой 2 — Data Engine

Ядро системы. Принимает события от источников, обрабатывает, выдаёт нормализованные метрики.

**Подробное описание:** см. `02-data-engine.md`

Ключевые функции:
- Нормализация и дедупликация данных
- Расчёт TPI-метрик с весами
- Построение графа коммуникаций
- Расчёт confidence scores
- Хранение в immutable data store
- Применение feedback loop от руководителя

---

### Слой 3 — Метрики

Стандартизированный JSON-пакет, который получает каждый агент:

```json
{
  "period": "2026-W11",
  "team_tpi": 6.8,
  "team_tpi_confidence": 0.87,
  "members": [...],
  "communication_graph": {...},
  "alerts": [...],
  "trends": {...}
}
```

**Confidence score** — уверенность в метрике от 0 до 1:
- `> 0.8` — данных достаточно, высокая точность
- `0.5–0.8` — данных средне, интерпретировать осторожно
- `< 0.5` — данных мало, метрика ненадёжна, агент должен это указать

---

### Слой 4 — Агенты

Пять агентов работают параллельно, каждый получает один и тот же JSON-пакет метрик, но фокусируется на своей области:

| Агент | Фокус | Выход |
|-------|-------|-------|
| HR-директор | Общая картина, финальный вывод | Недельный отчёт, приоритеты |
| Администратор | Качество данных, health check | Статус сбора, алерты |
| Социолог | Социодинамика, граф | Социограмма, опросы |
| Аналитик | Статистика, паттерны | Детальный анализ метрик |
| Коуч | Действия руководителя | План коучинга, подготовка к встречам |

**Важно:** агенты не хранят состояние между вызовами. Вся память — в Data Engine.

---

### Слой 5 — Рекомендации

Финальные артефакты для руководителя:

- **Дашборд недели** — структурированный HTML/Telegram-отчёт
- **Алерты** — экстренные сигналы при резком падении TPI
- **Коучинг-план** — конкретные разговоры, которые нужно провести
- **Управленческие инсайты** — паттерны, риски, возможности

## Улучшения архитектуры v2

### 1. Event-driven vs Polling

**Проблема v1:** система запрашивала данные по расписанию (cron), что создавало задержку до 24 часов.

**Решение v2:** event-driven архитектура
```
Событие в источнике → Webhook/Queue → Data Engine → Обновление метрики
```

- Telegram: webhook на каждое сообщение
- Zoom: webhook post-meeting
- CRM: вебхук при изменении сделки/задачи
- Отчёты: триггер при отправке формы

**Результат:** метрики актуальны с задержкой ≤ 5 минут вместо 24 часов.

---

### 2. Immutable Data Store

**Проблема v1:** данные перезаписывались, история терялась.

**Решение v2:** append-only хранилище
```sql
-- Данные только добавляются, никогда не изменяются
INSERT INTO events (timestamp, source, employee_id, type, payload)
-- Для корректировки создаётся новая запись с флагом correction
INSERT INTO events (timestamp, source, employee_id, type, payload, corrects_event_id)
```

**Результат:** полный аудит истории, возможность ретроспективного анализа, нет потери данных.

---

### 3. Agent Result Caching

**Проблема v1:** при каждом запросе агент вызывался заново, даже если метрики не изменились.

**Решение v2:** кеш результатов агентов
```json
{
  "agent": "analyst",
  "input_hash": "sha256:abc123...",
  "result": {...},
  "cached_at": "2026-03-16T10:00:00Z",
  "expires_at": "2026-03-16T22:00:00Z"
}
```

- При неизменных метриках возвращается кеш
- Инвалидируется при событиях с высоким приоритетом (алерты, новые данные)
- **Экономия:** ~70% LLM-вызовов

---

### 4. Confidence Scores для каждой метрики

**Проблема v1:** агент давал уверенный вывод даже при 1 сообщении за неделю.

**Решение v2:** каждая метрика содержит `confidence`:

```json
{
  "communication_score": 7.2,
  "communication_confidence": 0.91,
  "discipline_score": 5.0,
  "discipline_confidence": 0.45,  // мало данных!
  "discipline_note": "Только 2 точки данных за неделю"
}
```

Агент обязан в системном промпте учитывать confidence:
- `< 0.5` → указывать "данных недостаточно" в выводе
- `0.5–0.7` → интерпретировать осторожно
- `> 0.7` → стандартная интерпретация

---

### 5. Feedback Loop в Data Engine

**Проблема v1:** обратная связь от руководителя нигде не использовалась для улучшения алгоритма.

**Решение v2:** feedback попадает в Data Engine, а не в агентов

```
Руководитель корректирует оценку
       ↓
Data Engine записывает коррекцию
       ↓
Алгоритм калибрует веса метрик
  (если AI говорит 6, а руководитель ставит 8 — пересмотреть веса)
       ↓
Следующий расчёт использует откалиброванные веса
```

Хранение коррекций:
```json
{
  "week": "2026-W11",
  "employee_id": "emp_042",
  "metric": "communication",
  "ai_score": 5.5,
  "human_score": 7.0,
  "human_comment": "Личный разговор в пятницу, ситуация разрешена",
  "adjustment_applied": true
}
```

## Технический стек (рекомендуемый)

| Компонент | Технология |
|-----------|-----------|
| Data Engine | Python 3.10+ / FastAPI |
| Очередь событий | Redis Streams или RabbitMQ |
| Хранилище данных | PostgreSQL (immutable events table) |
| Кеш результатов | Redis |
| LLM | OpenAI GPT-4o или Anthropic Claude 3.5 Sonnet |
| Чат-интерфейс | Telegram Bot API (aiogram) |
| Визуализация | HTML + D3.js |

---

## Точки мониторинга

- Источники данных: статус последнего события (freshness)
- Data Engine: время расчёта метрик
- Агенты: latency LLM-вызовов, cache hit rate
- Confidence scores: средний confidence по команде (сигнал качества данных)
