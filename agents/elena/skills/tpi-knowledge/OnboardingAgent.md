# OnboardingAgent — Спецификация

> Агент первого контакта. Превращает "незнакомого клиента" в company_config.json
> который понимает вся система Елены.
> Работает один раз при подключении + при существенных изменениях в компании.

---

## Роль в архитектуре

OnboardingAgent — единственная точка, где система узнаёт о конкретном бизнесе.
После него все остальные агенты работают с нормализованными данными и не знают,
чей это клиент, какой у него продукт и как зовут сотрудников.

**Принцип:** задать минимум вопросов → получить максимум конфигурации.

---

## Структура интервью — 5 блоков, ~15 вопросов

### Блок A — Бизнес-контекст (3 вопроса)
Нужен для: сезонного коэффициента S(t), модели выручки, интерпретации потерь

A1. Как называется компания и чем занимается? (1–2 предложения)
A2. Как устроена выручка?
    [ ] Подписка (ежемесячный платёж)
    [ ] Транзакции (оплата за сделку)
    [ ] Проект (оплата по завершении)
    [ ] Смешанная модель
A3. Есть ли выраженная сезонность? Если да — когда пик, когда спад?

### Блок B — Команда (4 вопроса)
Нужен для: создания employee_registry, отделов, базовых ожиданий

B1. Перечисли сотрудников: Имя, должность, отдел, Telegram @username
B2. Кто руководитель? Кто принимает финальные решения по оценкам?
B3. Есть ли сотрудники на испытательном сроке? (→ онбординг-режим, мягкие пороги)
B4. Есть ли роли, где результативность не измеряется в деньгах напрямую?
    (контент, разработка, HR, операции) — если да, что является KPI этой роли?

### Блок C — Правила собственника (4 вопроса)
Нужен для: owner_rules в конфиге, настройки триггеров, весов дисциплины

C1. Какие ритуалы обязательны для всех?
    (ежедневный отчёт / утренний зум / еженедельный созвон / другое)
C2. Что для тебя "красная линия" — то, что ты никогда не терпишь в команде?
    (примеры: игнор руководителя, грубость с клиентом, срыв дедлайна без предупреждения)
C3. Что для тебя признак сильного сотрудника — помимо выполнения плана?
C4. Есть ли особые правила для конкретных ролей?
    (например: "разработчик может не писать ежедневный отчёт, если закрывает задачи в таскере")

### Блок D — Источники данных (3 вопроса)
Нужен для: конфигурации DataBridge, понимания что доступно сразу

D1. Какой мессенджер использует команда для работы?
    [ ] Telegram  [ ] Slack  [ ] Bitrix-чат  [ ] WhatsApp  [ ] Другой
D2. Есть ли CRM? Какая? Можно ли получить выгрузку план-факт по сотруднику?
D3. Есть ли таскер (трекер задач)?
    [ ] Notion  [ ] Jira  [ ] Trello  [ ] ClickUp  [ ] Google Таблицы  [ ] Нет

### Блок E — Финансовые данные (2 вопроса)
Нужен для: EVM, TCO, петли верификации

E1. Где хранятся данные по ФОТ? (1С / Excel / Google Таблица / в голове)
    Готов ли руководитель заполнять простую таблицу раз в неделю?
E2. Какова примерная текущая месячная выручка команды?
    (нужна для расчёта потерь — не для публичного отчёта)

## Протокол интервью

1. **Формат:** диалог в Telegram с OnboardingAgent (не форма, не опрос)
2. **Длина:** 20–30 минут, один сеанс
3. **Стиль агента:** задаёт один вопрос → ждёт → уточняет если ответ неполный
4. **Валидация:** после каждого блока агент резюмирует — "Я понял так: ..."
5. **Финал:** агент показывает company_config в читаемом виде → руководитель подтверждает
6. **Сохранение:** конфиг записывается в систему, OnboardingAgent передаёт управление оркестратору

## Что происходит после онбординга

```
company_config.json
       ↓
DataBridge читает конфиг → знает откуда брать данные
ChatWatcher читает конфиг → знает какой мессенджер, какие ритуалы
TPICalculator читает конфиг → знает owner_rules, пороги, веса
AlertAgent читает конфиг → знает red_lines, кому слать
EVMEngine читает конфиг → знает revenue_model, fot_source
ReportBuilder читает конфиг → знает имена, отделы, получателя
```

Каждый агент читает только свою секцию конфига. Ядро не меняется.

## Особые случаи

### Несколько команд / подразделений
Создаётся отдельный company_config для каждой команды.
Оркестратор может агрегировать TPI по нескольким конфигам → холдинговый дашборд.

### Нет CRM / нет таскера
Система переходит в "ручной режим":
- Руководитель заполняет Google Таблицу: `Неделя | Сотрудник | KPI_план | KPI_факт`
- DataBridge читает таблицу вместо API
- Это Starter-режим, не ограничение

### Смена сотрудника
OnboardingAgent запускается частично — только Блок B для нового человека.
Остальной конфиг не трогается.

### Смена бизнес-модели
Запускается полный повторный онбординг.
Старый конфиг архивируется, модель начинает накапливать новую baseline.
