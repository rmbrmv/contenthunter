# Anti-Patterns — Что НЕ работает

> Записывай сюда каждый негативный фидбек, правку или переделку.

## Формат записи

```
## YYYY-MM-DD: [Контекст задачи]
**Что не сработало:** [Описание]
**Почему:** [Причина провала]
**Урок:** [Что делать иначе в будущем]
```

---

## 2026-02-24: Чтение только первых 26 колонок таблицы

**Что не сработало:** Читала таблицу через диапазон A1:Z100 — пропустила все оценки (таблица идёт до колонки BZ, 73+ колонок).
**Почему:** Предполагала стандартный размер, не проверила реальную ширину.
**Урок:** Сначала читать строку заголовков через A1:BZ1, чтобы понять реальный размер таблицы.

## 2026-02-24: Включение клиентских встреч в анализ токсичности

**Что не сработало:** Включила встречи «Артём» и «Зум-встречи. Роман Абрамов (Дмитрий Кузьмин)» — пользователь сказал, что это клиентские, не внутренние.
**Почему:** Не проверила состав участников перед включением.
**Урок:** Внутренние встречи = только: Разработка, Кирилл, УК, Планерка ОП, Продакшен. Всё остальное — уточнять или исключать.

## 2026-02-24: Нестабильная методика анализа токсичности

**Что не сработало:** Менял методику 4+ раз (разные формулы, разные веса, разные источники) → каждый раз разные «самые токсичные» → Роман, потом Евгений, потом Павлов Иван → пользователь спросил «ты фантазируешь что ли?»
**Почему:** Не зафиксировала методику с самого начала. Каждая переделка = новый подход.
**Урок:** ОДИН раз определить формулу, зафиксировать в файле, больше не менять. Если методика меняется — объяснять явно.

## 2026-02-24: @ники и ссылки в отчётах

**Что не сработало:** Использовала @username и ссылки на Telegram в отчётах о токсичности.
**Почему:** Не учла, что отчёт может публиковаться.
**Урок:** В аналитических отчётах — только имена (или должности), без @ников, без ссылок.

## 2026-02-24: Включение фамилий в отчёты

**Что не сработало:** Использовала фамилии сотрудников в отчёте.
**Почему:** Не учла требование пользователя — только имена.
**Урок:** В публичных отчётах — только имя. Фамилии убирать.

## 2026-02-24: Анализ чатов без встреч

**Что не сработало:** Несколько раз давала отчёт по токсичности только на основе чатов, забывая про Zoom встречи.
**Почему:** Встречи дополнительный источник, про который легко забыть.
**Урок:** Анализ токсичности = ВСЕГДА чаты + встречи. Оба источника обязательны.

## 2026-03-03: Хардкод номеров строк в таблице

**Что не сработало:** Записала оценки по хардкодированным номерам строк (37=Геращенко, 38=Киркицкий и т.д.) — но строки в таблице сдвинулись. Оценки попали не тем людям: чужой человек (Ксения Гурьянова, стр 54) получила оценку 7, пустые строки (39, 49, 52) тоже получили оценки.
**Почему:** Запомнила номера строк из первичного чтения, не учла что таблица живая — строки вставляются и удаляются.
**Урок:** НИКОГДА не хардкодить номера строк. Алгоритм: 1) читать A1:BZ200, 2) искать сотрудника по имени в колонке F (индекс 5), 3) писать в найденную строку.

## 2026-03-03: Запись чисел как текст в Google Sheets

**Что не сработало:** Передавала значения как строки ('7', '6') с `valueInputOption: RAW` — в ячейках появился апостроф (текстовый формат), пользователь увидел символ и спросил зачем.
**Почему:** RAW + строка = текст в Sheets. Нужно либо передавать int, либо USER_ENTERED.
**Урок:** Всегда передавать числа как `int` и использовать `valueInputOption: USER_ENTERED`.

## 2026-03-02: Крон сработал, но сообщение не дошло

**Что не сработало:** Крон «Трэкшен: оценки команды» в 18:00 МСК сработал (статус ok), но пользователь не получил сообщение — спросил «чё не пришла» в 20:15.
**Почему:** Вероятно, проблема с доставкой в main session или канал.
**Урок:** Если крон пропустил — прийти самой когда пользователь напомнит. Проверить настройки delivery крона.

## 2026-03-06: Клиентские чаты не дают данных для TPI

**Что не сработало:** Ожидала, что новые клиентские чаты (Symmetry, Ambassadori, etc.) улучшат TPI-оценки. Данные почти не изменились (+57 строк из 3000+).
**Почему:** Сотрудники в клиентских чатах не пишут — там переписка с клиентами. Внутренняя активность в других чатах.
**Урок:** Для TPI важны внутренние чаты (разработка, фермы, поддержка), не клиентские. Клиентские = только для оценки внешней коммуникации.

## 2026-03-06: TPI дашборд — оформление отчёта

**Что не сработало:** Прямые цитаты из сообщений в колонке наблюдений; упоминание внутренних деталей («v3», «строгая методология», «baseline») в теле отчёта для руководителя.
**Почему:** Руководителю не нужны цитаты — нужно поведенческое саммари. Технические внутренние пометки засоряют отчёт.
**Паттерн/Урок:** В HR-отчётах — только саммари поведения («работает в выходные», «делится экспертизой»). Без версий, методологий, технических меток. Динамика (Δ к предыдущей неделе) обязательна в каждом дашборде.

## 2026-03-06: Оценка по должности вместо сигналов

**Что не сработало:** Первые оценки Попова Кирилла и Абрамова Романа — 8.0+ баллов по вовлечённости и экспертизе, обоснование фактически «CEO/co-founder, значит высокий». Роман указал на проблему.
**Почему:** Предположение что роль = сигнал. Методология этого не допускает.
**Урок:** Должность/роль — не сигнал. Только поведение с цитатой. Если цитат нет — baseline 5, не угадывать.

## 2026-03-09: Нерелевантный ответ на «почему ты молчишь»

**Что не сработало:** На вопрос «почему ты молчишь» ответила заготовленным сообщением про Google Sheets токен и таблицу «Трэкшен веры» — полностью не в контексте.
**Почему:** Видимо, сработал кешированный/шаблонный ответ из старого контекста про оценки. Не прочитала ситуацию.
**Урок:** Перед ответом на «почему ты молчишь» — проверить текущий контекст разговора. Не отправлять заготовленные блоки без проверки релевантности.

## 2026-03-06: Предположение «клиентские чаты дадут данные по сотрудникам»

**Что не сработало:** Ожидала, что после добавления 16+ клиентских чатов улучшатся TPI-оценки сотрудников — прирост оказался +57 строк из 3000+, почти ноль.
**Почему:** Сотрудники в клиентских чатах не общаются между собой — там переписка с клиентами. Внутренние сигналы поведения — только во внутренних чатах.
**Урок:** Клиентские чаты ≠ данные для внутренней оценки. Для TPI критичны: разработка, фермы, поддержка, внутренние операционные чаты. Перед добавлением нового источника — оценить, кто и как там пишет.

## 2026-03-03: Неполное планирование до старта задачи

**Что не сработало:** Начала записывать оценки в таблицу не проверив, что строки именно там — в итоге полный откат и переделка.
**Почему:** Не было чек-листа перед операцией записи: прочитать → найти → верифицировать → записать.
**Урок:** Для любой записи в таблицу: (1) читаю весь диапазон, (2) ищу строку по имени, (3) показываю план записи пользователю, (4) пишу, (5) верифицирую. Пять шагов, никаких сокращений.

## 2026-03-06: Оценка руководителей без сигналов = завышение

**Что не сработало:** Попов Кирилл и Абрамов Роман получили 8.0+ в первых итерациях, потому что «CEO/co-founder = высокая вовлечённость по умолчанию».
**Почему:** Нарушение базового принципа — оцениваем поведение, не роль.
**Урок:** Должность не является сигналом ни в каком направлении. Нет поведенческой цитаты → baseline 5. Даже для CEO.

## 2026-03-16: Еженедельный сбор данных не запущен проактивно

**Что не сработало:** В memory/2026-03-09.md был чёткий план — запустить collect_data.py в пятницу 13.03. Не запустила. Кирилл пришёл сам в понедельник утром, данных не было.
**Почему:** Нет механизма самоконтроля по задачам из memory. Читаю планы, но не отслеживаю их выполнение.
**Урок:** После записи плана в memory — добавлять напоминание в HEARTBEAT.md или крон. «Написала в memory» ≠ «сделала». Плановые пятничные задачи нужен крон, не просто запись.

## 2026-03-16: Отсутствие ежедневного memory-файла

**Почему:** Не создаю memory-файл в начале сессии / после значимых событий.
**Урок:** При первом значимом событии дня (запрос, задача, встреча) — сразу создавать/обновлять memory/YYYY-MM-DD.md. Не накапливать в голове.

## 2026-03-18: http.server как способ доставки файлов

**Что не сработало:** Подняла python3 -m http.server 8099, дала ссылку — Кирилл не смог открыть (порт закрыт файрволлом на сервере).
**Почему:** Внешний порт не проброшен, файрволл блокирует нестандартные порты.
**Урок:** http.server для раздачи файлов не работает без настройки файрволла. Надёжный путь — message(action=send, filePath=...) напрямую в Telegram.

## 2026-03-20: Плановый сбор данных снова не запущен (третий цикл)

**Урок:** Еженедельный сбор данных должен быть в HEARTBEAT.md как крон (пятница 17:00), а не в memory как «напоминание». Если нет крона — задача не будет выполнена.

## 2026-03-27: collect_data.py собирает промпты, не сохраняет TPI в БД

**Что не сработало:** Запустила collect_data.py, получила 3520 строк — и посчитала задачу выполненной. Но скрипт генерирует промпты для оценки субагентом, а не TPI-оценки. Без save_tpi.py данные не попадут в БД и дашборд 30.03 будет пустым.
**Почему:** Перепутала «данные собраны» и «TPI посчитан». Это два разных шага: collect → оценка субагентом → save_tpi.py.
**Урок:** Пятничный цикл = три шага, не один: (1) collect_data.py, (2) субагент оценивает по промптам, (3) save_tpi.py сохраняет в БД. Без шага 3 — дашборд в понедельник будет пустым.

## 2026-03-26: Урок зафиксирован, но не применён в тот же день

**Что не сработало:** 25.03 добавила в anti-patterns: «флаги без follow-up → написать Кириллу сама». 26.03 — follow-up не отправлен. Осознание ошибки не конвертировалось в действие.
**Почему:** Рефлексия происходит в конце дня (21:00). Урок записывается, но у него нет «триггера исполнения» на следующий день — нет задачи, нет крона, нет напоминания.
**Урок:** После записи анти-паттерна, требующего конкретного действия → сразу добавить в current-task.md или memory «Сделать завтра: [конкретное действие]». Иначе урок остаётся декларацией.

**Урок:** Если прошло 5 рабочих дней без ответа на переданный флаг → написать Кириллу или Роману коротко: «[имя], статус по [флагу]?». Не ждать планового дашборда. Не ждать вопроса.

## 2026-03-24: Флаги висят без follow-up — системная проблема

**Что не сработало:** Ревчук (с 06.03), Павлов Иван (с 26.02), Ежков (с 16.03) — флаги переданы руководству, но обратной связи нет. Я фиксирую «ждём действий Кирилла» неделями, не делая follow-up.
**Почему:** Нет процесса «нет ответа за X дней → напомнить». Передала — и считаю задачу выполненной.
**Урок:** Передача флага ≠ закрытие флага. Если нет ответа 5 рабочих дней → повторное упоминание в следующем дашборде или прямой вопрос руководителю. Флаг закрывается только когда есть явное решение или статус «принято к сведению».

## 2026-03-23: Memory-файлы не создаются в дни «без событий»

**Что не сработало:** За 21.03 и 22.03 нет memory-файлов. Рефлексия 22.03 прошла без дневного контекста — анализировала по артефактам, а не по живым записям.
**Почему:** Создаю memory-файл только когда «что-то случилось». Дни рефлексии, мониторинга и ожидания — тоже рабочие дни.
**Урок:** Memory-файл создавать каждый рабочий день, даже если событий мало. Минимум: дата, статус флагов, что мониторила. 5 строк лучше, чем пустота.

## 2026-03-22: Системный паттерн — память фиксирует, крон исполняет

**Что не сработало:** За три недели подряд (13.03, 20.03, и риск 27.03) еженедельный сбор collect_data.py не запускается. Причина устойчивая: задача пишется в memory, не в крон.
**Почему:** Memory = архив намерений. Крон = механизм исполнения. Они не равнозначны.
**Урок:** Для любой задачи с конкретным расписанием (еженедельно, ежедневно) — только крон, не memory. Запись в memory без крона = задача не будет сделана. Правило: «Если важно — крон. Если не задействован крон — сделано не будет».
