## autowarm — url-poller Instagram: фикс зависания awaiting_url (Генри, 2026-03-23)

**Проблема:** задачи Instagram зависали в статусе `awaiting_url` на часы/сутки.

**Причина:** Node.js `https.get` в `scrapeAllVideos()` получал 429 от Instagram из-за блокировки по TLS fingerprint. `curl` с теми же заголовками проходил нормально.

**Фикс (коммит `4a20727`):** Instagram-блок переписан на `curl` через `child_process.exec`:
- `User-Agent: Mozilla/5.0 (iPhone; ...) Instagram/311.0.0`
- `X-IG-App-ID: 936619743392459`

**Результат:** 3 зависших задачи (id 2, 6, 11, аккаунт inakent06) закрылись за 15 сек. Новые задачи получают URL ≤ 2 мин.

**Правило:** Instagram API в `server.js` — только через `curl`. `https.get`/`axios`/`fetch` → 429.

**Репо:** GenGo2/delivery-contenthunter

## validator — фикс автосохранения BrandPage (2026-03-23)

**Проблема:** на странице `/client/brand` изменения в полях `textarea` (`description`, `target_audience`, `usp`, `content_goals`, `stop_words`) не сохранялись. При переходе в другой раздел и возврате данные откатывались.

**Причина:** у `textarea` не было никаких обработчиков событий — ни `@blur`, ни `@input`. Только `input`-поля с `@blur="scheduleAutoSave"` работали корректно.

**Фикс:** добавлены `watch`-наблюдатели в `BrandPage.vue` с guard `if (!loading.value)`:
```js
watch(form, () => { if (!loading.value) scheduleAutoSave() }, { deep: true })
watch(keywordsInput, () => { if (!loading.value) scheduleAutoSave() })
watch(selectedVoices, () => { if (!loading.value) scheduleAutoSave() }, { deep: true })
watch(selectedFormats, () => { if (!loading.value) scheduleAutoSave() }, { deep: true })
watch(socialLinks, () => { if (!loading.value) scheduleAutoSave() }, { deep: true })
```

**Правило:** в Vue 3 для форм с `textarea` — всегда вешать `watch(form, ..., { deep: true })`, а не полагаться только на `@blur` на отдельных полях.

**Файл:** `workspace-genri/validator/frontend/src/pages/client/BrandPage.vue`

## unic-worker — фикс зависания задач в processing (2026-03-23, Генри)

**Проблема:** задачи зависали навсегда в статусе `processing` если все схемы падали с ошибкой.

**Причина:** `mark_task_error()` (строка 86) — asyncpg иногда возвращает JSONB-колонку как строку. Вызов `dict('{"a":1}')` итерирует по символам → `ValueError` → исключение проглатывалось → задача не переводилась в `error`.

**Фикс:** безопасный парсинг `meta`:
```python
raw_meta = row['meta'] if row and row['meta'] else {}
if isinstance(raw_meta, str):
    import json as _j; raw_meta = _j.loads(raw_meta)
meta = dict(raw_meta) if raw_meta else {}
```

**Файл:** `autowarm/unic-worker/worker.py` (на 91.98.180.103)
**Коммит:** `3352057` в GenGo2/delivery-contenthunter

**Диагностика зависших задач:**
```sql
SELECT id, input_video_name, current_status, error_message, schemes_done, schemes_error
FROM unic_tasks WHERE current_status = 'processing';
```
Processing дольше 10 мин = зависание. Сброс:
```sql
UPDATE unic_tasks SET current_status='pending', schemes_done=0, schemes_error=0, error_message=NULL WHERE id=<id>;
```
Затем `pm2 restart unic-worker` на `91.98.180.103`.

**Правило:** asyncpg JSONB → всегда проверять `isinstance(val, str)` перед `dict()`.
