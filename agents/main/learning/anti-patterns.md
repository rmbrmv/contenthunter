## Уроки из документации OpenClaw (14.03.2026)

### 1. agents.list — ОБЯЗАТЕЛЬНО для изоляции агентов
- Агенты БЕЗ записи в `agents.list` используют ДЕФОЛТНЫЙ workspace (`~/.openclaw/workspace`) → читают чужой SOUL.md
- Правило: каждый новый агент → `openclaw agents add <id> --workspace ~/.openclaw/workspace-<id>`
- Это добавляет запись в agents.list, создаёт agentDir и workspace автоматически

### 2. Named accounts НЕ наследуют от default account
- `channels.telegram.allowFrom` → только для default account
- Для каждого named account нужен свой `allowFrom` в account config
- `dmPolicy: "open"` → обязательно `allowFrom: ["*"]` в том же account config
- `dmPolicy: "allowlist"` → обязательно хотя бы один ID в `allowFrom`, иначе config validation error

### 3. Routing binding без accountId = только default account
- `{agentId: "x", match: {channel: "telegram"}}` → только default account
- Для всех аккаунтов: `accountId: "*"` (channel-wide fallback)
- Приоритет: peer match > parentPeer > guildId+roles > guildId > accountId > channel > default agent

### 4. Per-topic agent routing
- Можно маршрутизировать разные топики к разным агентам через `topics.<threadId>.agentId`
- Session key: `agent:<agentId>:telegram:group:<chatId>:topic:<threadId>`

### 5. groupAllowFrom отдельно от allowFrom
- Security boundary (2026.2.25+): group sender auth НЕ наследует DM pairing-store
- Для групп нужен отдельный `groupAllowFrom` или per-group `allowFrom`

### 6. Правильный способ создания агента
```bash
openclaw agents add <id> --workspace ~/.openclaw/workspace-<id>
openclaw channels add --channel telegram --account <accountId> --token <token>
openclaw agents bind --agent <id> --bind telegram:<accountId>
openclaw gateway restart
openclaw agents bindings  # проверить
```
